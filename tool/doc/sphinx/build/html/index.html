About¶

The WITSML Test Tool was developed to fulfill the following objectives:

  • Allow functional testing of a WITSML server
  • Enable testing GetFromStore-only servers
  • Simplify the preparation and maintenance of test scripts

Test Tool Packages¶

Certification tool consists of several packages

wtl - WITSML Test Tool

wsvt - WITSML Schema Validation Tool

wtools - WITSML dataset tools.

wtl Package (WITSML Test Tool)¶

This package doing actual job

witsml.py¶

The witsml.py file is the interface to the WITSML Test Library (WTL). It defines the script commands and other parameters to be used by the Test Scripts and Test Suites. The witsml.py file includes:

  • WITSML constant definitions
  • Script commands

Key specified values in the API and schema specification are defined in witsml.py to simplify the writing of the Test Scripts and to make them more readable. These constant definitions should not be changed by the Test Scripts. Constants names are mostly uppercase (case is preserved for names in the WITSML specification) with multiple words separated by underscores. The following are examples of constant definitions:

The witsml.py file defines a set of functions to be used as script commands. They define a minimum set of test primitives required to implement the WITSML server functional testing. Although the Test Scripts could use any Python code and any of the WTL functionality directly, it is recommended that the Test Scripts limit themselves for the most part to the use of the script commands defined in witsml.py and very basic Python constructs such as ‘if’, ‘for’, etc. This simplifies the test development, the readability of the Test Scripts and the maintenance of the tests. The script commands defined in witsml.py execute functionality implemented in the WTL, which automatically performs actions specified in the WITSML specification, thus relieving the Test Scripts from having to do this. wtools Package (WITSML dataset Tool)¶

this one does witsml dataset comparison

Comparison of two object use following logic:
  • we take two xmls and normalize them - we remove all nodes that should not be compared, or that added by server and can not be validated we arange all parts of WITSML object in same way, where order is not regulated by standard (for example curves) all values, are parsed and then printed back same way, so timezones will disapear, doubles will be truncated the same way.
-then not actual comparison done but encapsulation we basically check that nodes that we added are still there, but we allow server to add some other weired stuff ( like modification timestamp)

Comparison procedure¶

wtools.common.isXMLEncapsulateAnotherXML(inspected_tree, original_tree, inspected_xml, original_xml, depth=0)¶
Checks weather XML tree 1 encapsulates XML tree 2. It is essentially comparison of XML, It will succeed if:
  • every node of original tree is exists in inspected XML in same order (if it a text node then node text is matches)
  • every attribute of original node exists in inspected node.
  • comparison of subtree succeeded.
this will require that WITSML objects (XML’s) was normalized:
  • all nodes that allowed to be different are removed
  • all attributes that are optional should be Nulled.
  • all nodes that contain datetime values need to be normalized to same format (timezone)
  • all nodes that contain double values need to be rounded to specific comparison precission
